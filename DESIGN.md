## Section 1 ‚Äî Database Schema Justification

Every column in the `events` table exists for a specific reason. Columns that could not be justified were not added.

**`event_id UUID`** ‚Äî surrogate primary key. UUID chosen over BIGSERIAL because events are written by multiple agents potentially on multiple nodes; UUID generation requires no coordination.

**`stream_id TEXT`** ‚Äî the aggregate stream this event belongs to. Text rather than a foreign key to a streams table because streams are created implicitly on first append. Format: `{aggregate_type}-{aggregate_id}` e.g. `loan-APEX-0031`.

**`stream_position BIGINT`** ‚Äî position within the stream, zero-based. Combined with `stream_id` in the UNIQUE constraint, this is the OCC enforcement mechanism. Two agents attempting to write position 4 simultaneously will have exactly one succeed and one fail with a unique constraint violation.

**`global_position BIGINT GENERATED ALWAYS AS IDENTITY`** ‚Äî monotonically increasing global ordering across all streams. Generated by PostgreSQL, never by application code ‚Äî this guarantees no gaps and no conflicts. Used by the ProjectionDaemon to process events in arrival order.

**`event_version SMALLINT DEFAULT 1`** ‚Äî schema version of the event payload. Used by upcasters to transform old events to current schema on read without modifying stored data.

**`payload JSONB`** ‚Äî the event data. JSONB (binary JSON) chosen over JSON for indexing support and faster reads. NOT NULL ‚Äî an event without a payload is not an event.

**`metadata JSONB DEFAULT '{}'`** ‚Äî cross-cutting concerns: `correlation_id`, `causation_id`, `chain_hash`. Separate from payload so domain code never needs to know about infrastructure concerns.

**`recorded_at TIMESTAMPTZ`** ‚Äî wall clock time the event was written. `TIMESTAMPTZ` (with timezone) not `TIMESTAMP` ‚Äî all times stored in UTC, displayed in local timezone.

**The `event_streams` table** exists for two reasons: (1) it holds `current_version` which is the OCC lock target ‚Äî `SELECT ... FOR UPDATE` on this row serialises concurrent appends, and (2) it allows O(1) stream version lookup without counting events.

**The `outbox` table** implements the Transactional Outbox Pattern. The event insert and the outbox insert happen in the same database transaction. This guarantees that if an event is durably stored, the intention to publish it externally is also durably stored ‚Äî even if the process crashes between the DB commit and the Kafka publish.

## Section 2 ‚Äî Aggregate Boundary Decisions

The system uses 6 aggregate boundaries instead of 1. The alternative of one `LoanApplication` aggregate for everything was considered and rejected.

**Why not one aggregate?**

If all events ‚Äî credit analysis, fraud screening, compliance checks, agent sessions ‚Äî lived in `loan-{id}`, then every agent would compete for the same OCC lock. At 1,000 applications/hour with 4 agents each writing multiple events per application, the `OptimisticConcurrencyError` retry rate would be extremely high. Each retry requires a round trip to reload the stream, increasing latency and cost.

**The chosen boundaries:**

`loan-{id}` is the lifecycle stream ‚Äî it records only lifecycle transition events (`ApplicationSubmitted`, `CreditAnalysisRequested`, `ApplicationApproved`, etc.). It is written to at well-defined handoff points, not during agent processing.

`credit-{id}`, `fraud-{id}`, `compliance-{id}` are agent output streams ‚Äî each agent owns exactly one output stream. Two credit analysis agents racing (NARR-01) only compete for the `credit-{id}` lock, not the entire application.

`agent-{type}-{session}` is the Gas Town stream ‚Äî each agent session has its own stream. Agent node execution events are written here, never to the application streams.

**The coupling this prevents:** Without this boundary, a slow fraud detection agent holding the `loan-{id}` lock would block a credit analysis agent that needs to write to the same stream. With separate streams, they proceed in parallel.

## Section 3 ‚Äî Optimistic Concurrency Control Implementation

OCC is implemented using PostgreSQL row-level locking inside a transaction:

```sql
BEGIN;
SELECT current_version FROM event_streams 
WHERE stream_id = $1 FOR UPDATE;
-- check current_version == expected_version
-- if not: ROLLBACK, raise OptimisticConcurrencyError
INSERT INTO events (...) VALUES (...);
UPDATE event_streams SET current_version = current_version + N;
COMMIT;
```

The `FOR UPDATE` clause acquires a row-level exclusive lock on the `event_streams` row. A second concurrent transaction attempting the same will block at `SELECT ... FOR UPDATE` until the first commits or rolls back. The UNIQUE constraint on `(stream_id, stream_position)` provides a second layer of protection against race conditions.

**Why not application-level locking?** Database-level locks survive process crashes. If an agent crashes between reading the version and writing the event, the database lock is automatically released when the connection closes. Application-level locks (asyncio locks, Redis) do not survive process crashes.

**The double-decision test (NARR-01):** Two `CreditAnalysisAgent` instances read `stream_version("credit-APEX-0031") = 0` and both attempt `append(..., expected_version=0)`. Exactly one acquires the `FOR UPDATE` lock first, writes at position 1, and updates `current_version = 1`. The second acquires the lock, reads `current_version = 1`, sees `1 != 0`, raises `OptimisticConcurrencyError`, reloads the stream, and retries with `expected_version=1`.

## Section 4 ‚Äî Event Versioning and Upcasting Strategy

Events are immutable. When the schema of an event changes, old stored events are never modified. Instead, upcasters transform old events to the current schema on read.

**Two upcasters implemented:**

`CreditAnalysisCompleted v1 ‚Üí v2`: version 1 did not include `regulatory_basis` or `model_deployment_id`. Version 2 adds both. The upcaster sets `regulatory_basis=[]` and `model_deployment_id="legacy-pre-2026"` as safe defaults. The sentinel value `"legacy-pre-2026"` signals to downstream consumers that the model version is unknown, which is honest ‚Äî fabricating a specific version string would corrupt compliance audit trails.

`DecisionGenerated v1 ‚Üí v2`: version 1 did not include `model_versions`. The upcaster adds `model_versions={}`. This is safe because an empty dict correctly represents "model version not recorded" rather than inventing a false attribution.

**Upcaster immutability guarantee:** Upcasters are pure functions ‚Äî they take a payload dict and return a new payload dict. They never write to the database. The test `test_upcaster_does_not_mutate_original` verifies this: the original event dict is unchanged after calling `upcast()`.

**Chain upcasting:** The `UpcasterRegistry.upcast()` method applies upcasters in a while loop until the current version is reached. If an event needs to go from v1 ‚Üí v2 ‚Üí v3, both upcasters are applied in sequence automatically.

## Section 5 ‚Äî Projection Design and Consistency Model

The system uses three projections with different consistency requirements:

**`ApplicationSummary`** (SLO: <500ms lag) ‚Äî answers "what is the current state of application X?" This is the hot read path for the loan officer dashboard. It handles 15 event types and updates the `application_summary` table on every lifecycle transition. The 500ms SLO reflects that loan officers can tolerate a half-second delay between an event being committed and the dashboard refreshing.

**`AgentPerformance`** (SLO: <2s lag) ‚Äî tracks per-agent metrics: average confidence, cost, duration, approve/decline/refer rates. Updated on `CreditAnalysisCompleted`, `AgentSessionCompleted`, and `DecisionGenerated`. The 2s SLO reflects that this is a management dashboard, not a real-time operational view.

**`ComplianceAudit`** (SLO: <2s lag) ‚Äî temporal snapshots of compliance state. Implements `get_compliance_at(application_id, timestamp)` which returns the compliance verdict as it was at any historical point. This is required for regulatory audits. The `compliance_audit_view` table stores one row per `ComplianceCheckCompleted` event with `snapshot_at` as the timestamp key.

**Checkpoint consistency:** The projection daemon saves checkpoints in the same database as the projections. This ensures that if the daemon restarts, it resumes from the exact position where it last successfully processed events ‚Äî no events are processed twice and none are skipped.

## Section 6 ‚Äî Gas Town Pattern and Crash Recovery

The Gas Town pattern ensures every agent records its session start as the very first event, before loading any context or doing any work. This makes crash recovery deterministic.

**Why "Gas Town":** The name comes from the pattern requirement that an agent must establish its "base camp" (session start) before venturing into the work. If the agent crashes in the field, it can return to base camp and reconstruct exactly where it was.

**How it works:**

1. Agent starts ‚Üí immediately appends `AgentSessionStarted` to `agent-{type}-{session}` stream. This is the Gas Town anchor.
2. Agent executes nodes ‚Üí appends `AgentNodeExecuted` after each node completes successfully.
3. Agent crashes ‚Üí `AgentSessionFailed` is appended (if the crash is detected) or the stream simply ends.

**Recovery via `reconstruct_agent_context()`:**

```python
context = await reconstruct_agent_context(store, "fraud_detection", "APEX-0031")
# Returns:
# {
#   "session_id": "sess-fra-abc12345",
#   "last_successful_node": "load_document_facts",
#   "needs_reconciliation": True,
#   "context_source": "prior_session_replay:sess-fra-abc12345"
# }
```

The new agent session starts with `context_source = "prior_session_replay:{old_session_id}"` ‚Äî this is verifiable in the event stream, proving to auditors that a recovery occurred and which session was recovered from.

**NARR-03 (crash recovery test):** FraudDetectionAgent crashes after `load_document_facts`. A new session starts, replays the crashed session stream, resumes from `cross_reference_registry`. The fraud stream ends up with exactly ONE `FraudScreeningCompleted` event ‚Äî no duplicate analysis.
